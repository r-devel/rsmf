2. Bug triage
- Gabe has a patch for a bug that he'd like accepted into R before the patch release code freeze in a couple of days: https://bugs.r-project.org/show_bug.cgi?id=18930
- Bug reports with patches that Kevin would like reviewed:
    - https://bugs.r-project.org/show_bug.cgi?id=19061

* Draft triaging guidelines: https://cryptpad.fr/doc/#/2/doc/edit/8hoNOVRXP7vMH3FF2TCm2d9e/.
HT has tried to lay this out in a linear way. Guides on paths that the triager might take on a given bug. Should be a workable document, but there are some open questions. Guidelines based on the permissions that the triage team currently has. Some questions require further permissions - these conversations with R Core can happen at useR! 2026 in four weeks.

* Question for HT: now that she's on R Core, can she accept patches into R? HT: yes, she's comfortable with some small patches. Others she would need to speak to other R Core members about. In the first instance, for RSMF related fixes, she can talk with Simon and Peter. Also discuss considerations around accepting a fix if this could break anything on CRAN. There's always a risk that some package relies on broken behaviour. In relation to whether HT can accept Gabe's patch (https://bugs.r-project.org/show_bug.cgi?id=18930) the outcome of the discussion is yes, she can (the consensus is that no packages likely to rely on the broken behaviour)

* Comment from Hadley:  I’d urge the group to consider how we might reduce the cost of proposing a simple patch to R. It’s a very high bar to require that every patch runs R CMD check on every CRAN package
    * Agreement that it is unreasonable to expect every patch to be checked against all CRAN packages. Kurt runs these checks daily. Michael L points out that one role for mentors is to help mentees understand what level of testing is necessary for any given patch.
    * Peter D: think hard and try not to break anything - if you later find/get told that something did break (e.g. by Kurt or Brian), then can revert, or propose a better fix.

* Peter advises not to do too much at once. But small changes, fixing bugs, then Heather can go ahead with this. Michael L notes that Heather was elected to R Core because they trust her with the code base :) Jeroen feels we should really start practicing this to work on the patch backlog. Only by exercising the process we can finetune it and build trust.

* Backlog of bugs/patches. Sometimes the backlog is finding an R Core member with time. Sometimes there are specific R Core members who need consulting. Things get tricker when there are design decisions.
